# Multi-Agent Session Orchestrator

Design for running 2-4 Claude Code agents in parallel inside a single devcontainer, each with isolated filesystem and Docker services.

---

## Architecture

```
Host (EC2/VPS)
 └── DevPod devcontainer (single, SSH access)
      ├── Agent: planner    (tmux pane)
      ├── Agent: developer  (tmux pane)
      ├── Agent: qa         (tmux pane)
      ├── Docker socket → host Docker daemon
      │   ├── dev-1-postgres, dev-1-meilisearch, ...  (dev-1-net)
      │   ├── qa-1-postgres, qa-1-meilisearch, ...    (qa-1-net)
      │   └── (planner has no services)
      └── claude-peers broker (localhost:7899, SQLite)

/workspace/
  repos/
    myproject/                     # canonical full clone (reference repo, gc.auto=0)
  sessions/
    registry.json                  # session state, flock-protected
    dev-1/                         # session clone
      docker-compose.override.yml  # generated: remapped names/networks/volumes
      .session.env                 # generated: service hostnames, ports
      backend/.env                 # generated: project env with remapped hosts
    qa-1/
      ...
```

### Key principles

- **One devcontainer, N agents.** Agents are processes inside the same container (tmux panes, background jobs).
- **Shared Docker socket.** All agents control the host Docker daemon. Isolation is by naming convention, not enforcement. This is a trust boundary — any agent can `docker rm -f` any container.
- **Services reached by container name.** The devcontainer joins each session's Docker network. Services use their default internal ports (5432, 7700, 9000). No port offsetting for Docker services.
- **Application processes need unique ports.** Backend servers, frontend dev servers run directly in the devcontainer. Two Fastify servers can't both bind 0.0.0.0:3000. Port allocation handles this.
- **Cross-session access works by default.** The devcontainer is on all session networks, so any agent can connect to any session's services (e.g., QA inspects developer's Postgres).

## Session Configuration File

Project authors drop `.ws-sessions.yml` in repo root. Optional — planner profile and basic service prefixing work without it.

```yaml
# .ws-sessions.yml
compose_file: docker-compose.yml       # default: auto-detected

# Services that need per-session isolation
services:
  postgres:
    container_name: myapp-postgres     # original name in docker-compose.yml
  redis:
    container_name: myapp-redis
  meilisearch:
    container_name: myapp-meilisearch
  minio:
    container_name: myapp-minio
  mailpit:
    container_name: myapp-mailpit

# Network defined in docker-compose.yml
network:
  name: myapp-network                  # original network name
  external: true                       # whether original is declared external

# Volumes with explicit name: fields (unnamed volumes are auto-scoped by compose project)
volumes:
  pgdata:
    name: myapp-pgdata                 # original explicit name, if any

# Env files that reference services by container name
env_files:
  - path: backend/.env
    template: backend/.env.example     # source to copy from
    vars:
      DATABASE_URL: "postgresql://user:pass@{postgres}:5432/mydb"
      REDIS_URL: "redis://{redis}:6379"
      MEILI_URL: "http://{meilisearch}:7700"
      MINIO_ENDPOINT: "http://{minio}:9000"
      SMTP_URL: "smtp://{mailpit}:1025"
      PORT: "{backend_port}"

# Non-Docker processes that need port allocation
processes:
  backend:
    port_env: PORT                     # env var that controls the listen port
    default_port: 3000
  frontend:
    port_env: VITE_PORT
    default_port: 5173

# Bootstrap commands (run in session repo directory)
bootstrap:
  pre: "pnpm install"                  # before services start
  post: "pnpm db:migrate && pnpm db:seed"  # after services are healthy
  skip_scripts: ["scripts/dev.sh"]     # project scripts that hardcode container names
```

### What's auto-detected vs must-specify

| Auto-detected | Must specify |
|---|---|
| docker-compose.yml location (repo root) | service-to-env-var mapping (`vars` in `env_files`) |
| git root | which compose services need isolation |
| existing container names from compose file | process commands and their port env vars |
| | network name and whether it's `external: true` |
| | volumes with explicit `name:` fields |

Without `.ws-sessions.yml`, the orchestrator creates isolated repo clones (planner) and starts services with auto-prefixed container names. But env var substitution requires the config file.

## Generated Files

### docker-compose.override.yml

Remaps container names, network, and explicitly named volumes. For session `dev-1`:

```yaml
# AUTO-GENERATED by ws-session. Do not edit.
services:
  postgres:
    container_name: dev-1-myapp-postgres
  redis:
    container_name: dev-1-myapp-redis
  meilisearch:
    container_name: dev-1-myapp-meilisearch
  minio:
    container_name: dev-1-myapp-minio
  mailpit:
    container_name: dev-1-myapp-mailpit

volumes:
  pgdata:
    name: dev-1-myapp-pgdata

networks:
  myapp-network:
    name: dev-1-myapp-network
    external: false
```

Notes:
- `external: true` in the original is overridden to `external: false` so compose creates the session-scoped network instead of looking for a pre-existing one.
- Volumes without explicit `name:` in the original are auto-scoped by `COMPOSE_PROJECT_NAME` (e.g., `dev-1_pgdata`). Only volumes with explicit names need override.
- Services with `build:` directives get an additional `image: dev-1-<original-image-name>` to prevent tag collisions across sessions.

### .session.env

```bash
# AUTO-GENERATED by ws-session. Do not edit.
SESSION_ID=dev-1
SESSION_PROFILE=developer
SESSION_INDEX=0
SESSION_REPO=/workspace/sessions/dev-1
COMPOSE_PROJECT_NAME=dev-1

# Docker service hostnames (remapped container names)
WS_POSTGRES=dev-1-myapp-postgres
WS_REDIS=dev-1-myapp-redis
WS_MEILISEARCH=dev-1-myapp-meilisearch
WS_MINIO=dev-1-myapp-minio
WS_MAILPIT=dev-1-myapp-mailpit

# Process ports
PORT=3000
VITE_PORT=5173
```

### Project .env files

Generated from `env_files` config. Placeholders `{service_name}` replaced with session-prefixed container names, `{process_port}` with allocated ports.

## Port Allocation

Only for non-Docker processes (backend, frontend dev servers). Docker services use their default ports internally — no port allocation needed.

Scheme: `default_port + (session_index * 100)`

```
Session 0 (dev-1):  backend=3000, frontend=5173
Session 1 (qa-1):   backend=3100, frontend=5273
Session 2 (dev-2):  backend=3200, frontend=5373
```

This is the starting point. At creation time, the orchestrator checks `ss -tlnp` for conflicts and bumps to the next free port if needed. Actual assigned ports are stored in the registry.

## CLI Interface

```
ws-session create <session-id> --project <repo-url-or-path> [--no-services] [--ref <branch>]
ws-session destroy <session-id> [--keep-volumes]
ws-session list
ws-session status <session-id>
ws-session services <session-id> start|stop|restart
ws-session refresh <session-id>
ws-session exec <session-id> <command...>
ws-session env <session-id>
```

### `create`

1. Validate session ID (lowercase alphanumeric + hyphens, max 20 chars, unique)
2. Allocate index, compute port range, check for conflicts
3. If reference repo doesn't exist, clone to `/workspace/repos/<project>` with `gc.auto=0`
4. `git clone --reference /workspace/repos/<project> file:///workspace/repos/<project> /workspace/sessions/<session-id>`
5. If `--ref` given, checkout that branch
6. Read `.ws-sessions.yml` from clone (or use defaults)
7. Generate `docker-compose.override.yml`
8. Generate `.session.env` and project .env files
9. Unless `--no-services`: `docker compose -p <session-id> up -d --wait`
10. `docker network connect <session-network> $(hostname)`
11. Run `bootstrap.pre` if defined
12. Run `bootstrap.post` if defined (services are healthy from `--wait`)
13. Append session context to clone's CLAUDE.md (excluded via `.git/info/exclude`)
14. Update registry (under flock)
15. Print summary

### `destroy`

1. `docker compose -p <session-id> down -v` (or without `-v` if `--keep-volumes`)
2. `docker network disconnect <network> $(hostname)` (ignore errors)
3. `rm -rf /workspace/sessions/<session-id>`
4. Update registry (under flock)

### `refresh`

Re-reads `docker-compose.yml` and `.ws-sessions.yml` from the session's current branch. Regenerates override and .env files. Restarts services if the override changed. Use after switching branches.

### `exec`

Runs a command in session context:

```bash
ws-session exec dev-1 pnpm test
# Equivalent to: cd /workspace/sessions/dev-1 && source .session.env && pnpm test
```

### `list`

```
SESSION   PROFILE      STATUS    SERVICES  PORTS
dev-1     developer    running   5/5       backend:3000 frontend:5173
qa-1      no-services  running   -         backend:3100
plan-1    no-services  ready     -         -
```

## Filesystem Isolation

Each session gets a full independent git repo via `git clone --reference`:

```bash
git clone --reference /workspace/repos/myproject \
  file:///workspace/repos/myproject \
  /workspace/sessions/dev-1
```

The reference repo at `/workspace/repos/myproject` shares object storage with session clones (via git alternates). This saves significant disk — only working tree files and new objects are duplicated. The reference repo has `gc.auto=0` to prevent garbage collection from removing objects that session clones depend on.

Not worktrees — worktrees share `.git/` internals (refs, objects, config). One agent's `git gc`, `git branch -D`, or `git worktree prune` can corrupt other agents' state.

Not `--dissociate` — that copies all objects into each clone, defeating the disk savings from `--reference`. Since the orchestrator controls the reference repo and disables GC on it, shared alternates are safe.

## Cross-Agent Communication

### Service access

The devcontainer is connected to all session networks. Any agent can reach any session's services:

```bash
# QA agent inspects developer's database
psql "postgresql://easybiz:localdev@dev-1-myapp-postgres:5432/easybiz"
```

### claude-peers-mcp (optional)

Real-time messaging between Claude Code agents. Enabled via `WS_PEERS=1` environment variable (default: disabled).

When enabled:
- The orchestrator starts the broker on `localhost:7899` during the first `ws-session create` (if not already running)
- The broker is container-wide — one instance serves all sessions
- Each Claude Code agent registers automatically via the MCP server

Tools available to agents: `list_peers`, `send_message`, `set_summary`, `check_messages`.

**Prerequisites** (installed in devcontainer image when peers support is desired):
- Bun runtime
- claude-peers-mcp: `bun install -g claude-peers-mcp` (or clone to `/workspace/.tools/claude-peers-mcp`)
- MCP server registered: `claude mcp add --scope user --transport stdio claude-peers -- bun /path/to/server.ts`

**Launch flag**: Claude Code must be started with `--dangerously-load-development-channels server:claude-peers` for real-time message push. Without it, agents must manually call `check_messages`.

To enable, set in your shell profile or `.session.env`:

```bash
export WS_PEERS=1
```

The orchestrator checks this variable. When unset or `0`, the broker is not started and MCP registration is skipped. Sessions work normally without peer messaging — agents can still access each other's services directly.

## Agent Integration

### CLAUDE.md injection

On session creation, the orchestrator appends to the clone's CLAUDE.md:

```markdown
# Session Context (auto-generated by ws-session)
- Session: dev-1 (developer)
- Services: postgres=dev-1-myapp-postgres, meilisearch=dev-1-myapp-meilisearch, ...
- Backend: localhost:3000
- Do NOT run project bootstrap scripts directly (they hardcode container names).
  Use `ws-session exec dev-1 <cmd>` or source .session.env first.
- Other sessions: qa-1 (backend:3100)
```

This block is excluded from git via `.git/info/exclude` — it never appears in commits or diffs.

Agents MUST be started from the session directory (`cd /workspace/sessions/dev-1 && claude`) so Claude Code reads the session-aware CLAUDE.md.

## Project Bootstrap Integration

Projects have their own setup scripts that may hardcode container names or run destructive docker commands. The orchestrator does NOT run them automatically.

Strategy:
- `skip_scripts` lists scripts that are unsafe in multi-session context
- `pre`/`post` in bootstrap config provide equivalent safe commands
- `.session.env` is sourced before bootstrap commands, so env vars point to the right services
- If project scripts are refactored to use env vars instead of hardcoded names, they become session-safe and can be removed from `skip_scripts`

## Edge Cases

**Reference repo doesn't exist yet.** First `create` for a project clones to `/workspace/repos/<project>`, sets `gc.auto=0`, then clones from that.

**Compose services fail to start.** Session is created, marked `degraded` in registry. Agent can debug and restart with `ws-session services <id> restart`.

**Devcontainer rebuild.** Docker networks and containers survive on the host. The devcontainer loses network connections. `ws-session list` detects this and prints reconnect instructions.

**Disk space.** With `--reference` (no `--dissociate`), only working tree files are duplicated per session. For a 500MB repo, 4 sessions ≈ 600MB total (shared objects + 4 working trees), not 2GB.

**Port conflicts.** Detected at creation time via `ss -tlnp`. Orchestrator bumps to next free port.

**Stale sessions.** If Docker resources are gone but session directory exists, `ws-session list` shows `stale`. `destroy` cleans up filesystem. `services start` re-creates Docker resources.

**Branch switches.** After `git checkout` to a branch with different `docker-compose.yml`, run `ws-session refresh <id>` to regenerate override and restart services.

**Compose `extends`/`include`.** Not supported. Override only applies to the top-level compose file. Document as a limitation.

## What This Design Does NOT Do

- Manage tmux sessions or agent processes — that's the user's concern
- Auto-detect env var to service mapping — too fragile, explicit config required
- Enforce isolation — shared Docker socket means any agent can touch anything
- Support Windows or macOS containers — Linux devcontainers only
- Replace Docker Compose — it generates overrides, not alternative orchestration
- Run on multiple hosts — single devcontainer; use separate DevPod workspaces for multi-host
